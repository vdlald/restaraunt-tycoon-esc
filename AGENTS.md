# Repository Guidelines

## Project Structure & Module Organization

This is a Roblox restaurant-tycoon prototype synchronized with Roblox Studio through Rojo. Source is split by runtime:

- `src/client/` contains client scripts, input/state, and React UI (`src/client/ui/`).
- `src/server/` contains ECS world setup, systems, player data, pools, and gameplay modules. Add domain objects under `src/server/Game/<Domain>/`.
- `src/shared/` holds modules used by both runtimes, including networking (`Net.luau`), constants, and validation.
- `Packages/`, `ServerPackages/`, and `DevPackages/` are generated by Wally; do not edit them directly.
- `default.project.json` maps these folders into the Roblox data model. `RestaurantTycoon.rbxlx.rbxl` is the Studio place file.

## Build, Test, and Development Commands

- `wally install` installs dependencies declared in `wally.toml`.
- `rojo build -o "RestaurantTycoonEcs.rbxlx"` builds a Studio place from the project mapping.
- `rojo serve` starts live synchronization after opening the place in Roblox Studio.
- `stylua src` formats Luau source. Run it before submitting changes.
- `selene src` lints production source using Roblox and TestEZ globals.

Tools and pinned versions are defined in `rokit.toml`; install them with Rokit when they are not available locally.

## Coding Style & Naming Conventions

Write strict Luau: source modules begin with `--!strict`. Follow the existing StyLua style—tabs for indentation, trailing commas in multiline tables, and readable one-type-per-line declarations. Use PascalCase for module files and exported types (`BuildingSystem`, `PlayerView`), camelCase for locals and fields, and suffix entry scripts with `.server.luau` or `.client.luau`.

Keep client/server boundaries explicit. Put shared packet definitions in `src/shared/Net.luau`; validate client-originated actions on the server. Gameplay systems should operate through the Jecs `World` wrapper rather than bypassing ECS state.

## Testing Guidelines

TestEZ is the project test framework; its globals are configured in `.luaurc` and `testez.yml`. Add tests under `tests/` using `describe("Module", function()` and focused `it("does ...", function()` cases. There is no repository test-runner script yet: run the configured TestEZ suite through the local Roblox/TestEZ workflow, and lint production changes with `selene src`.

## Commit & Pull Request Guidelines

Recent history uses short, imperative subjects such as `fix`, `refactor`, and `fix typos`. Prefer a concise scope when useful, for example `building: validate rotation`. Keep commits focused. Pull requests should explain gameplay/runtime impact, list validation performed, link related issues, and include Roblox Studio screenshots or a short recording for visible UI or placement changes.
